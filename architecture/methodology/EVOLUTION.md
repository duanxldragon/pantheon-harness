# 方法论演化机制

> 方法论不是写一次就完事的，它应该随实践持续进化。
> 本文档定义"什么时候改、怎么改、谁批准"。

## 版本规则

| 字段 | 文件 | 含义 |
|---|---|---|
| 版本号 | `pantheon-harness/VERSION` | semver：MAJOR.MINOR.PATCH |
| 变更日志 | `pantheon-harness/CHANGELOG.md` | 人类可读的变更历史 |

### 版本号含义

| 变更类型 | 升级位置 | 示例 |
|---|---|---|
| 新增方法论章节、模板、规则 | MINOR | 1.0 → 1.1 |
| 修正措辞、链接、格式 | PATCH | 1.0.0 → 1.0.1 |
| 推翻核心原则、重写模型 | MAJOR | 1.x → 2.0 |

## 演化触发条件

修改方法论文档的触发条件（满足任意一条即可）：

1. **实践触达**：同一个问题在 3 次不同任务中反复出现
2. **复盘驱动**：月度复盘发现流程缺陷
3. **工具变更**：Codex/OMX/Claude Code 版本大升级导致旧流程不适用
4. **人为反馈**：你明确说"这个流程需要改"

## 演化流程

```
遇到问题/发现改进点
  │
  ├─ 记录到 project-memory（即时）
  │
  ├─ 累计 3 次同类问题？
  │   └─ 是 → 编写方法论补丁
  │
  ├─ 月度复盘发现？
  │   └─ 是 → 列入下月改进计划
  │
  └─ Claude 整理 → 更新文档 → 更新 VERSION → 更新 CHANGELOG
```

## 复盘节奏

| 频率 | 内容 | 产出 |
|---|---|---|
| **每周** | 快速回顾：本周有什么新发现？哪个流程不顺？ | 1-2 条记忆条目 |
| **每月** | 系统性复盘：对照复盘模板填写 | 方法论改进清单 |
| **每季度** | 大版本升级：重新审视三权分立、质量门禁是否仍然适用 | MAJOR 版本号升级决策 |

## 反馈入口

日常反馈不需要等复盘。直接告诉 Claude：
- "记住，以后遇到 XX 情况时用 YY 方式处理"
- "上次 ZZ 流程太慢了，精简一下"

Claude 会自动写入 `memory/` 并在下次复盘时提炼为方法论变更。
