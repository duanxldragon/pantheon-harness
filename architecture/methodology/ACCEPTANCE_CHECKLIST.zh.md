# 非程序员验收清单

> 你不会写代码，但你可以用这份清单判断 AI 产出的代码是否合格。
> 验收时对照逐项检查，全部通过 = 可以合并。

## 使用说明

每次 Generator 完成一个任务、Reviewer 审查通过后，你打开这份清单：
- ✅ = 通过
- ❌ = 不通过（记录具体原因）
- ⏭️ = 不适用（注明原因）

---

## 1. 功能验证（必查）

| # | 检查项 | 怎么查 | 结果 |
|---|---|---|---|
| 1.1 | 功能是否按需求描述的那样工作？ | 打开页面/调用 API，走一遍主流程 | |
| 1.2 | 边界情况是否处理？（空列表、无权限、异常输入） | 故意触发空数据、错误输入 | |
| 1.3 | 操作后数据是否正确持久化？（创建/编辑/删除） | 刷新页面，确认数据还在/已变更 | |
| 1.4 | 没有"点了没反应"的按钮或链接？ | 每个按钮点一遍 | |

## 2. 视觉质量（有 UI 改动时必查）

| # | 检查项 | 怎么查 | 结果 |
|---|---|---|---|
| 2.1 | 有没有渲染截图/浏览器证据？ | 看 evidence 里的截图或浏览器记录 | |
| 2.2 | 页面布局是否整齐？（没有明显的错位、重叠、溢出） | 肉眼检查截图 | |
| 2.3 | 文字是否完整显示？（没有截断、乱码、中英混排问题） | 检查所有可见文字 | |
| 2.4 | 按钮、表格、表单风格是否和系统其他页面一致？ | 对比其他管理页面 | |
| 2.5 | loading 状态是否正常？（数据加载时有骨架屏或 loading 提示） | 刷新页面看加载过程 | |
| 2.6 | 空数据状态是否正常？（无数据时有提示，不是空白页） | 查一个空列表/空详情 | |

## 3. 错误与反馈（必查）

| # | 检查项 | 怎么查 | 结果 |
|---|---|---|---|
| 3.1 | 操作成功后有提示吗？（toast/消息条） | 创建、编辑、删除后观察 | |
| 3.2 | 操作失败后有明确的错误提示吗？（不是白屏或 500） | 故意输入非法数据 | |
| 3.3 | 表单必填项为空时，提交按钮是否拦截并提示？ | 空表单点提交 | |
| 3.4 | 网络超时或接口报错时，页面是否有重试按钮？ | 断网后操作 | |

## 4. CI/CD 门禁（必查）

| # | 检查项 | 怎么查 | 结果 |
|---|---|---|---|
| 4.1 | CI 所有检查是否全部通过？（绿色 ✅） | 打开 GitHub PR 页面看 Checks | |
| 4.2 | Backend Tests 是否通过？ | CI 日志 | |
| 4.3 | Frontend Contract 是否通过？ | CI 日志 | |
| 4.4 | Smoke Sanity 是否通过？ | CI 日志 | |
| 4.5 | Quality Gates 是否通过？ | CI 日志 | |
| 4.6 | SonarQube 是否有新增问题？ | SonarCloud 页面 | |

## 5. PR 审查确认（必查）

| # | 检查项 | 怎么查 | 结果 |
|---|---|---|---|
| 5.1 | Reviewer 是否已完成代码审查并给出通过结论？ | 看 review artifact | |
| 5.2 | PR 描述是否完整？（改了什么、怎么验证的） | 阅读 PR body | |
| 5.3 | Commit 信息是否符合规范格式？ | 检查 commit 列表 | |
| 5.4 | 改动范围是否和任务要求一致？（没多改没少改） | 对比任务描述和 diff | |

## 6. 文档与副作用（改了什么就查什么）

| # | 检查项 | 怎么查 | 结果 |
|---|---|---|---|
| 6.1 | 如果改了接口：API 文档是否同步更新？ | 检查 docs/contracts | |
| 6.2 | 如果加了菜单：权限配置是否正确？ | 用非管理员账号测试 | |
| 6.3 | 如果加了页面：i18n 多语言 key 是否完整？ | 切换语言查看 | |
| 6.4 | 如果改了数据库：迁移脚本是否存在且可回滚？ | 检查 migrations/ 目录 | |

## 快速验收流程（日常使用）

```
1. 打开 PR 页面
   ├─ 看 CI 是否全绿？ → 不全绿 → 退回给 Dispatcher 处理
   └─ 全绿 → 继续

2. 看 Review artifact
   ├─ 有 ❌ 吗？ → 有 → 退回给 Generator 修
   └─ 全部 ✅ → 继续

3. 看截图/浏览器证据（UI 改动时必须）
   ├─ 看起来对吗？ → 不对 → 说具体哪里不对
   └─ 对 → 继续

4. 走一遍主要功能流程
   ├─ 正常流程跑通了吗？ → 不通 → 记录复现步骤
   └─ 通 → 合并 PR ✅
```

---

## 遇到问题怎么反馈

反馈时不要只说"有问题"，用这个模板：

```
问题描述：  [一句话描述]
复现步骤：  1. 打开 XX 页面 2. 点击 XX 按钮 3. 看到 XX 错误
预期行为：  应该 XX
实际行为：  实际 XX
截图：      [有就贴]
```

这样 Dispatcher 可以直接转化为 Generator 的修复指令，不需要你再解释一遍。
